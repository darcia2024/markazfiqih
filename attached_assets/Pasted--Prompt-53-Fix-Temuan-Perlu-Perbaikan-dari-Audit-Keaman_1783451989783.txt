## Section 2 — Fix Race Condition Voucher (`used_count`)

Edit `supabase/functions/checkout/index.ts`.

Ganti pola read-then-write untuk validasi & increment voucher (baca
`used_count`, cek `< max_uses`, lalu update terpisah) dengan atomic
update. Buat fungsi Postgres berikut lewat SQL manual dan panggil lewat
`.rpc()` dari edge function:
```sql
CREATE OR REPLACE FUNCTION increment_voucher_usage(voucher_id UUID)
RETURNS TABLE(id UUID) AS $$
BEGIN
  RETURN QUERY
  UPDATE class_vouchers
  SET used_count = used_count + 1
  WHERE class_vouchers.id = voucher_id
    AND (max_uses IS NULL OR used_count < max_uses)
  RETURNING class_vouchers.id;
END;
$$ LANGUAGE plpgsql;
```
Kasih tahu saya (Darcia) bahwa SQL ini perlu dijalankan manual di Supabase
SQL Editor — cantumkan di ringkasan akhir.

Lalu di edge function, ganti pola read-then-write dengan:
```ts
const { data: voucherUpdateResult, error: voucherUpdateError } = await supabaseAdmin
  .rpc('increment_voucher_usage', { voucher_id: voucherId });

if (voucherUpdateError || !voucherUpdateResult?.length) {
  return new Response(
    JSON.stringify({ error: 'Kode voucher sudah tidak berlaku atau kuota habis.' }),
    { status: 400, headers: corsHeaders },
  );
}
```

## Section 3 — Pindahkan Increment `used_count` ke `simulate-success`

Masih di alur voucher: `used_count` sekarang di-increment saat **checkout**
(saat invoice dibuat), bukan saat **pembayaran benar-benar sukses**. Kalau
user tidak menyelesaikan pembayaran, voucher tetap terpakai sia-sia.

- Di `supabase/functions/checkout/index.ts`: HAPUS logic increment
  `used_count` dari sini (voucher tetap divalidasi di sini untuk
  menghitung harga, tapi TIDAK di-increment).
- Di `supabase/functions/simulate-success/index.ts`: TAMBAHKAN logic
  untuk increment `used_count` pada voucher yang dipakai invoice ini
  (gunakan fungsi `increment_voucher_usage` yang sama dari Section 2),
  dilakukan bersamaan dengan proses lain di situ (update invoice jadi
  `paid`, buat enrollment). Perlu mengetahui voucher mana yang dipakai —
  cek apakah ada cara untuk menyimpan referensi voucher pada invoice atau
  invoice_items saat checkout (mungkin perlu menambah kolom `voucher_id`
  pada `invoices` atau `invoice_items` supaya `simulate-success` tahu
  voucher mana yang harus di-increment; kalau kolom ini belum ada,
  laporkan ke saya perlu migrasi tambahan sebelum bisa
  mengimplementasikan section ini dengan benar).

## Section 6 — Bersihkan Policy Duplikat di `user_profiles`

Jalankan dulu untuk melihat semua policy yang ada:
```sql
SELECT policyname, cmd FROM pg_policies WHERE tablename = 'user_profiles' ORDER BY cmd, policyname;
```
Untuk setiap operasi (SELECT/INSERT/UPDATE) yang punya lebih dari satu
policy dengan fungsi yang sama, hapus yang duplikat dan sisakan satu saja
(perhatikan: untuk UPDATE, pastikan yang disisakan adalah policy yang
sudah diperbaiki di SQL fix kritis sebelumnya, jangan sampai menghapus
yang benar). Laporkan hasil pembersihan.
